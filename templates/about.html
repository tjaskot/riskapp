Notes:
Our outputs are expected to be very linear in the data sets because the information we predict is all based on trend data. If a regression model is to be interpolated, then one must leverage gathered data from devlopment scrum team and input the respective values into a combined prediction chart of the three sets of charts this application develops. For example, once a trend of stability is produced by a contract, then the average cost per unit of work can be accurately predicted. Once this prediction model is created, the user can then create a repeating table with available tasking and bandwidth with precise numbers based on trend information.
Because the amount of release hours per sprint are either static or vary with littel fluxuation, it is clear to the observer that releasing more often will produce better results for the customer as well as the developers. The developers will be able to review their work and make revisions with a quicker turnaround as compared to a sprint with longer weeks and iterations. The product manager and customer will be able to provide more precise feedback because their observatiions are more frequent and include more communication.
The percentage of success for each value were defined by correlating integer values to the respective english phrases. For example the input for tech writer "What is the percentage of effor your Tech Writer..." is interpolated as integer success value of 95%. This was a project decision, as writing documents has litle failure. This logic applies to all of the input enligh phrases which correlate to integer values.
Each individual group gets 30 points allocated to it because there are 6 groups total and with there being 180 degrees in a dial chart 180/6 is equal to 30. The algorithm then takes each area into account and subtracts points accordingly with the weighted values for each group.
For security purposes, the variables located in the javascript files are open and do not matter if their data is available. However, variables located in the python file are automatically handled by flask and not readily available to a user. Therefore all sensitive data for this application is located in the riskapp.py file.
The ultimate cost variable is calculated by the median salary for each individual and accounts for the total number of sprints. The risk variable is calculated by the percentage of success rate for each given group of individual. The shcedule variableis calculated based on the number of weeks for each sprint iteration cycle and takes into account the risk calcualtion because taking time out of the schedule to hit a deadline inherintly assumes risk, and as that risk is accepted into the program or application the schedule should reflect that accomodation. The schedule variable is also based off the success of the testers' capabilities to debug all items to find defects.
